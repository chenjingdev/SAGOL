## Qualitative assessment

Both responses are **competent and address the same root cause set.** Neither dominates.

**Where full wins (marginal):**
- More fix proposals (5 vs 3) — covers "lint rule to enforce timezone in deps" and explicit UTC-storage pattern
- Cleaner short/medium-term split in fix recommendations
- Slightly more code examples

**Where stripped wins (marginal but surprising):**
- Catches `TzContext` Provider `value` memoization issue that full variant omits. This is a subtle but commonly overlooked React gotcha. Full version never mentions it.
- A/B/C structure is arguably more consumable than full's 5-item numbered list
- Confirmation plan is more concrete (4 specific steps vs 3 open questions)

**Tool usage anomaly:**
Stripped variant used ~2.7x more tokens and ~2x more tool calls despite smaller input context. Hypothesis: stripped agent did more exploration/searches (probably tried to read the referenced `.tsx` files to compensate for missing report detail). Full agent committed to a textual diagnosis faster.

This is itself interesting: a stripped context may trigger MORE tool-use behavior as the model tries to recover missing detail — partially undoing the token savings.

## Signal interpretation

At 2% context reduction on a ~150K context, **no clear quality degradation from stripping**, and a faint hint that stripping may free cognitive space for secondary concerns (Provider memoization catch).

This is **consistent with** — but not strong evidence for — the user's hypothesis that stripping helps at longer context. At this context size (150K) and reduction ratio (2%), effect is too small to separate from natural response variance.

## What this does NOT tell us

1. Whether stripping helps at 300K+ context (user's hypothesized threshold)
2. Whether effect scales with reduction ratio (would bigger reports produce clearer signal?)
3. Whether effect appears on reference or detail follow-up tasks (independent follow-up was the easiest test case)
4. Whether the tool-use anomaly reproduces or is noise

## Caveats about subagent simulation

- Subagents are *general-purpose Claude Code agents*, not a precise simulation of the original session environment
- Both had access to Read/Write tools — the stripped variant appeared to lean on them more
- Text-only constraint may have forced artificial terseness in both
- Neither response had tool-result feedback loops that would exist in real agentic work

## Recommendations

Options ordered by information-gain per effort:

1. **Scale reduction ratio** — run a second case with a much larger synthetic report (15-20K chars, ~10-15% of context). If stripping is net-positive, larger reduction should amplify the signal.
2. **Scale context** — run a case at ~300K context (user's hypothesized threshold). Requires splicing multiple real sessions as filler.
3. **Scale N** — run 5+ cases at current scale to disentangle signal from variance.
4. **Change followup type** — run reference/detail followup cases. Detail followup is the expected NEGATIVE control (stripping should hurt); if it does not, something is wrong with the test design.
5. **Stop** — faint signal is consistent with "maybe real, maybe not, effect likely too small to justify engineering at this context scale."

My recommendation if continuing: **(1) bigger report + (4) detail followup control case.** Those two together give us the clearest next signal with minimal extra work.
