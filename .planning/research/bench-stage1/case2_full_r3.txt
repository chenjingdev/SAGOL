코드를 직접 못 보는 상태에서 추정하면, 가장 흔한 원인 2가지입니다.

**원인 1: DatePicker가 TzContext를 구독하지 않음**

DatePicker 내부에서 `useContext(TzContext)` 또는 커스텀 훅(`useTz` 등)으로 현재 timezone을 가져오고 있지 않을 가능성. 초기 렌더 시점에 timezone을 한 번 읽고 로컬 state에 저장해서, context가 바뀌어도 re-render가 트리거되지 않는 패턴입니다.

```tsx
// 문제 패턴
const DatePicker = () => {
  const { timezone } = useTz();
  const [tz, setTz] = useState(timezone); // 초기값만 캡처, 이후 변경 무시
  // ...
};
```

**Fix:** `useState`로 timezone을 복사하지 말고, context 값을 직접 사용하거나, `useEffect`로 context 변경을 감지해서 동기화.

```tsx
// 수정
const DatePicker = () => {
  const { timezone } = useTz(); // 직접 사용, state 복사 X
  // 또는
  useEffect(() => { setTz(timezone); }, [timezone]);
};
```

**원인 2: date 라이브러리 인스턴스가 timezone을 캐싱**

`dayjs`, `luxon`, `date-fns-tz` 같은 라이브러리를 쓸 때, timezone이 설정된 인스턴스를 컴포넌트 외부(모듈 스코프)에서 한 번만 생성하는 경우. 예를 들어:

```tsx
// 문제 패턴 (모듈 스코프)
const formatter = dayjs.tz.setDefault('Asia/Seoul'); // 앱 시작 시 1회만
```

**Fix:** formatter나 date 인스턴스를 컴포넌트 내부에서 timezone 변경 시마다 재생성. `useMemo`에 timezone을 dependency로 넣으면 됩니다.

```tsx
const formattedDate = useMemo(
  () => dayjs(value).tz(timezone),
  [value, timezone]
);
```

**추가 가능성:**

- DatePicker가 제3자 라이브러리(react-datepicker 등)를 래핑하고 있으면, 해당 라이브러리의 locale/timezone 설정이 prop 변경에 반응하지 않는 경우도 있음. 이때는 `key={timezone}`을 DatePicker에 줘서 강제 remount하는 게 가장 깔끔.
- TzContext의 Provider 자체가 값을 제대로 업데이트하는지도 확인 필요. `value`에 새 객체를 넘기지 않고 mutate만 하면 React가 변경을 감지 못함.

**확인 순서:**
1. `DatePicker.tsx`에서 `useTz` 또는 `useContext(TzContext)`를 어떻게 쓰는지
2. timezone 값이 useState에 복사되는지
3. date 라이브러리 인스턴스의 생성 위치 (모듈 스코프 vs 컴포넌트 내부)
4. TzContext Provider의 value 업데이트 방식
