파일을 직접 확인할 수 없는 상태이므로 컴포넌트 이름과 구조 기반으로 원인을 추정합니다.

## 원인 추정

`DatePicker`가 timezone 변경에 즉시 반응하지 않는 전형적인 패턴은 3가지입니다:

### 1. 초기화 시점에만 timezone을 읽는 문제 (가능성 높음)

`DatePicker.tsx`에서 `TzContext`의 timezone 값을 **마운트 시점**에만 읽어서 내부 state에 복사해두는 패턴:

```tsx
// 문제 패턴
const { timezone } = useTzContext();
const [displayDate, setDisplayDate] = useState(() =>
  convertToTz(value, timezone)  // 마운트 시점 timezone으로 고정
);
```

이 경우 context가 바뀌어도 `useState` 초기값은 다시 평가되지 않으므로 화면이 갱신 안 됩니다.

### 2. useEffect 의존성 배열에서 timezone 누락

```tsx
useEffect(() => {
  setDisplayDate(convertToTz(value, timezone));
}, [value]);  // ← timezone이 빠져있음
```

`value`가 바뀔 때만 재계산하고, timezone 변경은 무시됩니다.

### 3. 서드파티 date picker 라이브러리의 key 문제

`react-datepicker` 같은 라이브러리를 쓰는 경우, 내부적으로 locale/timezone 관련 값을 인스턴스 생성 시에만 세팅합니다. timezone이 바뀌어도 인스턴스가 재생성되지 않으면 반영이 안 됩니다.

## Fix 방향

**A. 파생 상태로 전환 (1, 2번 원인일 때)**

timezone을 별도 state에 복사하지 말고, 렌더링마다 직접 계산:

```tsx
const { timezone } = useTzContext();
const displayDate = useMemo(
  () => convertToTz(value, timezone),
  [value, timezone]
);
```

또는 `useEffect` 의존성에 `timezone` 추가:

```tsx
useEffect(() => {
  setDisplayDate(convertToTz(value, timezone));
}, [value, timezone]);
```

**B. key 기반 강제 리마운트 (3번 원인일 때)**

서드파티 라이브러리가 timezone 변경을 내부적으로 처리 못 하면, `key`에 timezone을 넣어서 강제 재생성:

```tsx
<ThirdPartyDatePicker
  key={timezone}  // timezone 바뀌면 인스턴스 재생성
  value={displayDate}
  ...
/>
```

**C. TzContext 자체 문제 확인**

`TzContext.tsx`에서 `value`를 매번 새 객체로 만들고 있으면 불필요한 리렌더가 생기고, 반대로 `value`가 memoize되지 않는데 하위 컴포넌트가 `React.memo`로 감싸져 있으면 변경이 전파가 안 될 수 있습니다. Provider의 value가 `useMemo`로 안정화되어 있는지 확인 필요.

## 확인 순서

1. `DatePicker.tsx`에서 `useTzContext()` 반환값을 어떻게 사용하는지 (state 복사 vs 직접 사용)
2. `useEffect` 의존성 배열에 timezone이 포함되어 있는지
3. 서드파티 date picker를 쓰는지, 쓴다면 어떤 라이브러리인지
4. `TzContext.tsx`의 Provider value가 올바르게 memoize되어 있는지

파일 내용을 보여주시면 정확한 원인과 패치를 짚어드리겠습니다.
